* Elpaca update broke package bootstrap defaults                   :emacs:elpaca:use-package:regression:
:PROPERTIES:
:DATE: 2026-03-01
:END:

After image + Emacs package updates, startup failed with:
- ~Cannot load evil/general/git-auto-commit-mode~
- ~use-package: Unrecognized keyword: :general~
- Elpaca dependency failures showing built-in ~transient 0.7.2.2~ below required minimums.

** Root cause
Current ~elpaca-use-package~ no longer honors ~elpaca-use-package-by-default~.
Config still set that variable, so most ~use-package~ forms stopped being Elpaca-managed.
That prevented key packages from being ensured, causing cascading parse/load failures.

** Fix
In Emacs init, replace legacy default with:
- ~use-package-always-ensure t~ inside the ~elpaca-use-package~ bootstrap block.

Minimal working block:
#+begin_src emacs-lisp
(elpaca elpaca-use-package
  (elpaca-use-package-mode)
  (setq use-package-always-ensure t)
  (setq elpaca-queue-limit 20))
#+end_src

** Verification
Batch startup succeeds without the previous cascading errors when forcing
~use-package-always-ensure t~.

** Key files
- ~/home/tsb/.config/emacs/init.el~
- ~/home/tsb/.config/emacs/modules/bootstrap.el~

** Notes
This failure mode can masquerade as a ~general.el~ issue, but root cause is
loss of Elpaca default package ensuring.

* Ollama vs llama.cpp inference overhead                                :ollama:llama-cpp:gpu:benchmarks:
:PROPERTIES:
:DATE: 2026-04-04
:PROMPT: ollama vs llama.cpp inference overhead benchmarks 2025 - is there actual measured tokens/sec difference once model is loaded?
:END:

** Summary
- Go HTTP layer overhead is trivial (~microseconds per token). NOT the bottleneck.
- Well-controlled single-user benchmarks show ~8-13% overhead for token generation, shrinking to ~5% when context size settings are matched.
- Main culprits: default context size divergence (Ollama defaults larger) and Ollama's new native engine being slower than llama.cpp for some architectures.
- Ollama collapses under concurrency (5 concurrent: 8 t/s vs llama.cpp 25 t/s on RTX 4090).
- For single-user interactive use, the difference is imperceptible (26 vs 28 t/s).
- Set ~OLLAMA_LLAMA_CPP=1~ to force llama.cpp backend if Ollama's native engine is slow on your model.

** The Go HTTP layer is NOT the bottleneck
Serializing JSON through Go adds ~0.1-0.5ms per token. At 30 t/s each token takes ~33ms — HTTP overhead is noise. Real gaps come from:
1. Default context size (Ollama defaults 2048-8192, llama.cpp defaults 512)
2. Ollama's native engine reimplementing model architectures slower than llama.cpp
3. Ollama's request scheduler degrading under concurrency

** Benchmark: RTX 4090, LLaMA 3.1 8B Q4_K_M ([[https://ventusserver.com/vs-ollama-inference-speed/][Ventus Servers]])
- Single request: llama.cpp 28.4 t/s vs Ollama 26.1 t/s (*~8%*)
- 5 concurrent: llama.cpp 25.2 t/s vs Ollama 8.3 t/s (Ollama collapses)
- Caveat: context sizes differed (32k vs 11k)

** Benchmark: Apple Silicon, Qwen 3.5 9B Q4_K_M ([[https://docs.bswen.com/blog/2026-03-15-llamacpp-faster-than-lmstudio-ollama/][BSWEN]])
- Mismatched defaults: llama.cpp 4.6 t/s vs Ollama 3.6 t/s
- Matched settings (ctx=4096, FA on): llama.cpp 4.2 t/s vs Ollama 3.6 t/s (*~14%*)
- Root cause: context size divergence dominates

** Benchmark: Raspberry Pi 5, TinyLlama 1.1B Q4_0 ([[https://medium.com/@omkar121212/ollama-vs-llama-cpp-on-raspberry-pi-5-8e7fbeb310de][Medium]])
- Token gen: llama.cpp 14.4 t/s vs Ollama 13.75 t/s (*~4.5%*)
- Prompt eval: llama.cpp 108.1 t/s vs Ollama 80.34 t/s (~26% — prompt processing overhead larger)

** Benchmark: M4 Pro 64GB, DeepSeek V3 Q4_K_M ([[https://macgpu.com/en/blog/2026-mac-inference-framework-vllm-mlx-ollama-llamacpp-benchmark.html][macgpu.com]])
- Single user: Ollama 58 t/s vs llama.cpp 52 t/s (*Ollama 12% FASTER*)
- Caveat: Ollama v0.8+ may use MLX backend on Apple Silicon — different inference path

** Ollama's native engine problem (the REAL issue)
Ollama has been building its own inference engine reimplementing model architectures.
For some models this is dramatically slower:
- Qwen3.5 27B on RTX 3090: Ollama native 12 t/s vs llama.cpp 35-40 t/s ([[https://github.com/ollama/ollama/issues/14579][#14579]])
- Qwen3.5 35B MoE: Ollama native 89 t/s vs =OLLAMA_LLAMA_CPP=1= 218 t/s — 2.4x gap ([[https://github.com/ollama/ollama/issues/14861][#14861]])

* Is Pyroscope SOTA for automatic project profile snapshots?     :pyroscope:profiling:observability:jolo:
:PROPERTIES:
:DATE: 2026-05-02
:PROMPT: lets $j-research and see if pyroscope is SOTA for automatically generating snapshots of our code in projects.
:END:

** Summary
- If "snapshot" means an on-demand runtime performance profile/flamegraph tied to project/commit/source lines, Grafana Pyroscope is a SOTA OSS choice in 2026, especially because Pyroscope 2.0 just shipped with lower-cost storage/query architecture and OTLP Profiles support.
- For Jolo projects, the target shape is not "profile everything all the time by default"; it is "profiling is always available": an agent can trigger a profile capture from any project, the artifact lands in the shared perf hub, and the user can inspect it in the same UI.
- The pragmatic split is: keep cheap on-demand Speedscope/profile artifacts in early templates, and make Pyroscope the optional host-side perf-hub service for persistent profile history, source-code drilldown, and commit-window comparison.
- Pyroscope's GitHub source integration is useful but not universal: source-line integration is documented for Go, Java, and Python, requires GitHub-hosted code, and needs ~service_repository~ plus ~service_git_ref~ labels.
- Auto-instrumentation is credible but permission-heavy. Grafana Alloy's eBPF profiler can collect profiles without app code changes, but it needs root/host PID namespace or Linux capabilities that do not fit ordinary locked-down devcontainers.
- OpenTelemetry Profiles reached public Alpha in March 2026. That confirms continuous profiling is becoming a first-class observability signal, but OTel Profiles is not yet the stable backend choice for critical production workloads.

** Findings
- Grafana documents Pyroscope as an OSS continuous profiling aggregation system that lets you analyze workload resource usage down to source line level and correlate profiles with metrics, logs, and traces ([[https://grafana.com/docs/pyroscope/latest/][Grafana Pyroscope docs]]).
- Pyroscope 2.0 was announced 2026-04-21. Grafana says v2 removes write-path replication, makes query workers stateless, supports OTLP profiling, and has been running in Grafana Cloud Profiles since April 2025 ([[https://grafana.com/blog/pyroscope-2-0-release/][Pyroscope 2.0 announcement]], [[https://grafana.com/docs/pyroscope/latest/release-notes/v2-0/][v2.0 release notes]]).
- Collection paths are flexible: a local Pyroscope server can be started as one process/container, but collecting app profiles still requires either SDK instrumentation or Grafana Alloy ([[https://grafana.com/docs/pyroscope/latest/get-started/][Get started with Pyroscope]]).
- Auto-instrumentation via Alloy is the closest match to "automatic snapshots." The ~pyroscope.ebpf~ component profiles the current host and supports native code plus common high-level runtimes, but requires root in the host PID namespace or a set of BPF/perf/ptrace-style capabilities ([[https://grafana.com/docs/alloy/latest/reference/components/pyroscope/pyroscope.ebpf/][Alloy pyroscope.ebpf docs]]).
- Pyroscope's profile-type support varies by collection method. Auto instrumentation covers CPU profiles; SDKs give richer runtime-specific profile types for some languages. The official support table lists SDK coverage for Go, Java, .NET, Ruby, Python, Rust, and Node.js ([[https://grafana.com/docs/pyroscope/latest/configure-client/profile-types/][Profile types and instrumentation]]).
- Python scaffolds can integrate directly with ~pyroscope-io~ by adding a small setup block with ~application_name~, ~server_address~, optional ~sample_rate~, and tags. That is easy enough for opt-in project templates, but it is still application instrumentation, not zero-touch capture ([[https://grafana.com/docs/pyroscope/latest/configure-client/language-sdks/python/][Pyroscope Python SDK docs]]).
- Go can be pulled via pprof endpoints by Alloy, which fits services that already expose ~net/http/pprof~. That is less invasive than SDK push mode but still requires exposing endpoints and running a collector ([[https://grafana.com/docs/pyroscope/latest/configure-client/grafana-alloy/go_pull/][Go pull-mode profiling]]).
- Source-code integration is a strong reason to prefer Pyroscope over raw Speedscope files for a host perf hub: Pyroscope can map profile frames back to GitHub source and show line-level resource breakdowns. The documented requirements are Go/Java/Python support, GitHub access, and labels including ~service_repository~ and ~service_git_ref~ ([[https://grafana.com/docs/pyroscope/latest/view-and-analyze-profile-data/line-by-line/][Pyroscope source-code integration]]).
- The source integration fetches code from GitHub at a git ref; it does not archive profile artifacts by itself. The perf hub still needs a ledger/artifact store keyed by project, run ID, commit, route, and testbed.
- Parca remains the closest OSS alternative for zero-instrumentation continuous profiling. Its agent uses eBPF, discovers Kubernetes/systemd targets, emits pprof profiles, and supports broad language coverage; but it is more infrastructure-profile-first and less aligned with the existing Grafana/VictoriaMetrics/k6 stack ([[https://github.com/parca-dev/parca-agent][Parca Agent]], [[https://github.com/parca-dev/parca][Parca]]).
- OpenTelemetry Profiles entered public Alpha in March 2026, with an OTLP profile data model, pprof interoperability, and an eBPF profiler. The same announcement says production-ready backends have not yet emerged and Alpha should not be used for critical production workloads ([[https://opentelemetry.io/blog/2026/profiles-alpha/][OpenTelemetry Profiles Alpha]]).

** Open Questions
- Should Pyroscope live only in the global perf hub, or should generated projects include opt-in SDK glue for Python/Node/Go so profile capture works even without host eBPF privileges?
